# 운트임 리포트·대시보드 섹션 템플릿

포스텔러(Forceteller)류 **읽기 쉬운 정보 설계**를 참고하되, **운트임 철학(엔진 근거·코칭·단정 금지)** 을 우선한다.  
이 문서는 UI/카피/프롬프트 작성 시 공통 레퍼런스다.

---

## 1. 브랜드 축 (변하지 않음)

| 원칙 | 실무 체크 |
|------|-----------|
| **근거** | 문장·카드는 `analyze` / `analyze_full` 산출, 또는 그걸 가공한 `saju_overview`·`pillars`·`monthly_fortune` 등 **필드와 연결** 가능해야 한다. |
| **코칭** | 한 줄 요약 → 근거 → **지금 할 수 있는 작은 행동** 순이 기본. |
| **표현** | 가능성·경향. 미래·타인·건강·법·투자 **단정** 금지. 공포·자극 카피 금지. |
| **투명성** | 생시는 양력·음력 표기 정책과 일치(입력 요약에 양·음 병기 등). |

---

## 2. 카드(섹션) 단위 공통 골격

레퍼런스 앱처럼 **주제별 카드**로 나눌 때, 카드마다 아래 순서를 맞춘다.

```
[카드 제목]  ← 생활어 우선, 필요 시 괄호로 전문 용어
한 줄 요약  ← 1문장, 스캔용
근거        ← 엔진 필드 인용 또는 요약에 실린 간지·십성·운 등 1~2개
본문        ← 2~4문장, 가능성 중심
실천/주의   ← 불필요하면 생략 가능하나 상담형은 짧게 권장
```

- **점수·등급·아이콘**은 보조. 카드의 결론을 점수가 대신하지 않게 한다.
- **한자만 덩어리**로 두지 말고, 같은 화면에 **한글 라벨**을 병기한다.

---

## 3. 엔진 필드 ↔ 섹션 매핑 (프론트·리포트 공통)

아래는 `frontend/src/types/report.ts` 및 `/api/analyze` 응답 구조와 대응한다.

| 영역 | 권장 섹션 제목 톤(예) | 주요 소스 필드 |
|------|----------------------|----------------|
| 사주 한눈에 | 원국 요약, 오행 균형 | `sajuOverview.pillars.*`, `fiveElements.counts`, `yinYangCounts` |
| 일간·십성 | 오늘의 관점, 질문 주제별 성향 | `pillars.day`, `analysis.sipsin`, `sajuOverview` 각 기둥 `sipsin` |
| 십이운성·신살 | 흐름·이슈 포인트 | `twelve`, `shinsal`, `analysis.twelve_fortunes`, `analysis.shinsal` |
| 공망·원국 | 빈 기운, 주의 맥락 | `gongmang.*`, `engineVoidBranches` |
| 대운 | 10년 단위 흐름 | `sajuOverview.daewoon`, `daewoon` (raw) |
| 월운/세운 | 월별 카드 | `monthlyFortune.months[]`: `narrative`, `flow`, `good`, `caution`, `action`, `stemTenGod`, `twelveStage`, `daewoonPillar`, `seunPillar` 등 |
| 절기 메타 | 월주 경계 설명 보조 | `calcMeta.monthTerm`, `monthTermTimeKst` |
| 통합 요약 | 전체 흐름 문단 | `unified.summary`, `analysis.*.summary` (있을 때) |

**규칙:** 화면에 넣는 문장마다 “이게 `raw`의 어느 블록에서 왔는지” 추적 가능한 형태가 이상적이다.

## 4. 제목 패턴 예시 (복붙용 아님 · 톤 참고)

- 원국: `오행이 모이는 곳`, `일간이 보여 주는 기본 태도`
- 운: `이번 달 흐름 한 줄`, `대운이 바꿀 수 있는 것`
- 관계: `사람 앞에서 나오는 나`, `말보다 먼저 나가는 반응`
- 실천: `지금 한 주에 쓸 수 있는 행동`

과장·낚시형(“반드시”, “절대”)은 쓰지 않는다.

---

## 5. AI 상담 (`counsel_service` 시스템 프롬프트)과의 정렬

상담 답변은 반드시:

1. **① 공감 → ② 해석 → ③ 원인 → ④ 제안**  
2. 요약에 없는 구체적 미래·사실 단정 금지  
3. 엔진 요약 인용 우선  

UI에서 “짧은 카드”로 보여 줄 때도 같은 순서를 유지하면 서비스 전체 톤이 맞는다.

---

## 6. 포스텔러 류에서 **차용하는 것만**

- 짧은 제목 + 짧은 단락, 주제별 카드 그리드  
- 생활어 라벨 + (전문 용어) 병기  
- 단계적 깊이(요약 → 상세)

**차용하지 않는 것:** 타 서비스 문구 복제, 근거 없는 공통 운세, 점수만으로 결론 내리기.
